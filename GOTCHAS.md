# Gotchas — 근본 원인 기록

> 우회/해결 전에 근본 원인을 먼저 기록한다 (Gotcha-First).

## 1. serverless-http는 바이너리 응답을 기본으로 손상시킨다 (2026-07-13)
- **증상**: 로컬(node server.js)에서 정상인 PNG가 Lambda 프로덕션에서 깨진 이미지로 렌더.
  응답은 200 + `image/png`인데 바이트가 106,033→191,767로 부풂, `file` 판정 "data".
- **근본 원인**: serverless-http가 binary 설정 없이 응답 본문을 UTF-8 텍스트로 인코딩 —
  Lambda 응답 경로에서 바이너리가 손상된다. 상태 코드·Content-Type만 보면 정상으로 착각.
- **해결**: `serverless(app, { binary: ['image/*'] })`.
- **교훈**: 배포 검증에서 상태 코드만 보지 말고 **바이트(md5)까지 대조**할 것. 정적 에셋을
  추가하는 WO는 Lambda 경유 응답의 무결성 검증을 완료 기준에 포함.

## 2. 손상된 응답은 수정 배포 후에도 브라우저 캐시에 살아남는다 (2026-07-13)
- **증상**: 서버 바이트를 고쳤는데(원본 md5 일치) 기존 방문자 브라우저에서 여전히 깨진 로고.
  새 탭에서도 재현 (HTTP 캐시는 탭 간 공유).
- **근본 원인**: express.static의 ETag는 파일(size+mtime) 기준이라 손상 전후가 동일 →
  재방문 시 304 Not Modified → 캐시된 손상 본문을 계속 사용.
- **해결**: 에셋 URL에 버전 쿼리(`?v=2`)로 캐시 키 자체를 변경.
- **교훈**: 잘못된 응답이 나간 에셋은 서버 수정만으로 끝나지 않는다 — URL 버전을 올려라.

## 3. Hermes TUI의 유휴 판정은 힌트 바 텍스트가 아니라 경과 타이머(⏱)로 (2026-07-13)
- **증상**: tmux 전송 가드가 유휴 상태에서도 영구 대기 → 타임아웃.
- **근본 원인**: 하단 힌트 바("msg=interrupt · /queue …")는 유휴/작업 중 상시 표시되는데,
  busy 패턴에 "interrupt"를 넣어 항상 busy로 오판. 반대로 진짜 busy 신호는
  상태 줄의 경과 타이머(`⏱3m 49s`)와 "Compacting context" — 활성 턴에만 나타남.
- **해결**: 유휴 = 프롬프트(❯) 존재 AND (⏱·Compacting 부재).
- **교훈**: TUI 상태 판정은 "상시 요소"와 "상태 의존 요소"를 실측으로 구분한 뒤 패턴을 잡을 것.
